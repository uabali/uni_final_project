================================================================================
                     RAG CHATBOT - AGILE SPRINT PLANI
================================================================================

Amac: Mevcut RAG projesini, 12GB VRAM (RTX 5070 Ti) kisitina uygun sekilde,
Ollama-tabanli local LLM, LangGraph agent, ve ses destegi ile production-ready
bir chatbot'a donusturmek.

================================================================================
                         MEVCUT DURUM ANALIZI
================================================================================

Proje su an fonksiyonel bir RAG pipeline: PDF/TXT ingestion, Qdrant vectorstore,
3 LLM backend (OpenAI/vLLM/Trendyol), hybrid search (Vector+BM25), auto strategy,
multi-query, reranking, CLI + Streamlit UI, benchmarking.

Tespit edilen problemler:
- src/llm.py: vLLM in-process yukleniyor (LangChain VLLM wrapper). Bu yaklasim
  12GB VRAM'de embedding + LLM + reranker'i ayni anda tasimakta zorlanir
- src/vectorstore.py: Debug log kodlari production kodun icine gomulu (~100 satir
  gereksiz log)
- Docker Compose yok, Qdrant manuel baslatiliyor
- Conversational memory yok (chatbot icin kritik eksik)
- Agent altyapisi yok
- Ses destegi yok

================================================================================
                    VRAM BUTCESI (RTX 5070 Ti - 12GB)
================================================================================

Ollama'ya gecis sonrasi hedef VRAM dagitimi:

  Bilesen                                  VRAM
  ---------------------------------------- ----------
  Ollama LLM (Q4_K_M quantized 8B)        ~5.0 GB
  Embedding (BAAI/bge-m3)                  ~1.5 GB
  Reranker (bge-reranker-base)             ~0.4 GB (lazy load)
  CUDA overhead                            ~1.0 GB
  Buffer (STT/TTS icin kullanilabilir)     ~4.1 GB
  ---------------------------------------- ----------
  TOPLAM                                   ~12.0 GB

Ollama avantaji: Model idle kaldiginda VRAM'i otomatik bosaltir
(OLLAMA_KEEP_ALIVE), bu sayede STT/TTS ile VRAM paylasimi mumkun.

================================================================================
               TEKNOLOJI KARARLARI VE GEREKCELERI
================================================================================

1. LLM BACKEND: OLLAMA (vLLM yerine)
-------------------------------------

Neden vLLM degil?
- vLLM'in gercek avantaji (continuous batching, PagedAttention) multi-user/
  high-concurrency senaryolarda ortaya cikar. Tek kullanici + 12GB GPU'da
  bu avantaj pratikte yok
- vLLM kurulumu 1-3 saat, Ollama <15 dakika
- vLLM 16GB+ VRAM'de optimal. 12GB'da model + KV cache + embedding icin sikisik
- Mevcut kodda langchain_community.llms.VLLM kullaniliyor - bu wrapper modeli
  Python process icinde yukluyor, dolayisiyla embedding modeli ile ayni GPU
  memory space'i paylasiyor
- Ollama ayri process olarak calisir, OpenAI-compatible API sunar, otomatik
  VRAM yonetimi yapar

Benchmark verileri:
- Tek kullanici: Ollama vs vLLM arasinda %4-6 fark (Ollama lehine)
- TTFT: vLLM 10.7ms vs Ollama 65ms (RAG icin ihmal edilebilir, retrieval ~100ms)
- Ollama 128 concurrent request'te coker ama bu senaryo 12GB tek GPU icin
  zaten gecersiz

Model onerisi: llama3.1:8b-instruct-q4_K_M veya qwen2.5:7b-instruct-q4_K_M
(Turkce icin Qwen daha basarili)


2. VECTOR STORE: QDRANT (mevcut, dogru secim)
----------------------------------------------

Qdrant zaten dogru secim. Nedenleri:
- Production parity (local = deployed ayni mimari)
- Advanced filtering (payload indexing)
- Rust tabanli performans
- Multi-vector destegi (ileride multimodal icin)
- ChromaDB daha basit ama production'a geciste migration gerektirir
- Milvus overkill (milyarlarca vektor icin)

Iyilestirme: Docker Compose ile tum stack'i birlestirmek


3. CHUNKING: RECURSIVE (mevcut) + MARKDOWN-AWARE iyilestirme
-------------------------------------------------------------

- Mevcut recursive chunking (600 char, 100 overlap) makul
- Semantic chunking ek complexity getiriyor, kazanim tartismali
- Late chunking trend ama uzun-context embedding modeli gerektiriyor
  (bge-m3 8192 token destekliyor, uygulanabilir)
- Oneri: Once recursive'i markdown-aware hale getir (MarkdownTextSplitter),
  sonra late chunking dene


4. RETRIEVER: SADELESTIRME GEREKLI
------------------------------------

Mevcut src/retriever.py 419 satir ve asiri karmasik. Auto strategy + dynamic K
+ hybrid + multi-query + reranking hepsi ic ice. Bu:
- Debug'i zorlastiriyor
- Her degisiklikte yan etki riski
- Yeni feature eklemek icin cesaret gerekiyor

Oneri: Retriever'i pipeline pattern ile sadelestir, her asamayi bagimsiz
middleware olarak yaz.


5. AGENT FRAMEWORK: LANGGRAPH
-------------------------------

- Zaten langgraph dependency'de var ama kullanilmiyor
- LangChain ekosistemi ile dogal entegrasyon
- ReAct pattern: Reason -> Act -> Observe -> Repeat
- State management, checkpointing, persistence built-in
- Tool calling: web search, calculator, document search, file operations


6. SES MODELLERI
-----------------

STT (Speech-to-Text): Faster-Whisper Large V3 Turbo
- ~4.7-10.2GB VRAM (batch size'a gore)
- 76x realtime speed
- Ollama idle iken VRAM bosaltir, STT kullanilabilir
- Alternatif: whisper.cpp (daha az VRAM, CPU-friendly)

TTS (Text-to-Speech): Kokoro TTS
- 82M parametre, ~350MB (quantized ~80MB)
- CPU'da bile calisir, GPU gerektirmez
- Apache 2.0 lisans
- 54 ses, FastAPI ile OpenAI-compatible endpoint
- Turkce destegi sinirli, ancak multilingual support gelisiyor

Alternatif TTS: Orpheus TTS
- 3B parametre (Llama-3b tabanli)
- Duygusal konusma (<laugh>, <sigh> tag'leri)
- ~200ms streaming latency
- Daha fazla VRAM gerektiriyor (~2-3GB Q4)
- Ingilizce odakli

================================================================================
              GUNCEL TEKNOLOJI HARITASI (2026 ITIBARIYLE)
================================================================================

RAG EVRIMI
-----------
- Naive RAG (chunk -> embed -> retrieve -> generate): Mevcut yaklasim
- Advanced RAG (query transform + rerank + hybrid): Mevcut yaklasim (iyi noktada)
- Modular RAG (pipeline pattern, her adim degistirilebilir): Sprint 2 hedefi
- Agentic RAG (agent karar verir: arama gerekli mi, hangi tool?): Sprint 4 hedefi
- Graph RAG (knowledge graph + vector search): Ileri seviye, su an gerekli degil

LLM SERVING (LOCAL)
--------------------
- Ollama: Tek kullanici, kolay kurulum, 12GB GPU icin ideal. ONERILEN SECIM
- vLLM: Multi-user, high-throughput. 16GB+ VRAM ve concurrent users icin
- llama.cpp: En dusuk seviye, maximum kontrol. Ollama zaten bunu kullaniyor
- LM Studio: GUI tabanli, developer-friendly ama scriptable degil
- LocalAI: OpenAI drop-in replacement, multi-model

EMBEDDING MODELLERI
--------------------
- BAAI/bge-m3: Mevcut secim, SOTA multilingual, 1024-dim. Dogru secim
- nomic-embed-text: Ollama native, daha hafif ama bge-m3 kadar basarili degil
- jina-embeddings-v3: Yeni nesil, task-specific, 8192 token context

VECTOR DB
----------
- Qdrant: Mevcut secim, production-ready, Rust-based. Dogru secim
- ChromaDB: Prototipleme icin, production'da Qdrant tercih edilmeli
- Milvus: Enterprise, milyarlarca vektor
- pgvector: PostgreSQL native, eger zaten Postgres kullaniyorsan

AGENT FRAMEWORK
----------------
- LangGraph: LangChain native, ReAct, state machines, persistence. ONERILEN
- CrewAI: Multi-agent, role-based teams
- AutoGen: Microsoft, coding agents
- Smolagents: HuggingFace, lightweight agents

SES MODELLERI (STT)
--------------------
- Faster-Whisper Large V3 Turbo: En hizli acik kaynak STT, 76x realtime. ONERILEN
- Whisper.cpp: Daha az VRAM, C++ tabanli
- Deepgram Nova-3: Cloud, en hizli ama parali

SES MODELLERI (TTS)
--------------------
- Kokoro TTS: 82M param, Apache 2.0, CPU-friendly, 54 ses. ONERILEN (baslangic)
- Orpheus TTS: 3B param, duygusal konusma, <laugh> tag'leri. ILERISI ICIN
- XTTS v2 (Coqui): Voice cloning, multilingual, daha agir
- Piper TTS: Cok hafif, edge devices icin

DIGER TRENDLER
---------------
- MCP (Model Context Protocol): Anthropic'in standardi, tool/resource paylasimi
- Late Chunking: Chunk'lamadan once tum dokumani embed et, sonra bol
- Contextual Retrieval: Chunk'a ust-baglam metadata ekle (Anthropic)
- ColBERT / ColPali: Token-level interaction, daha iyi retrieval ama agir
